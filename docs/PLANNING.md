# App Factory — 기획 & 전략 문서

> 1인 개발자의 유틸 앱 **양산 파이프라인** 프로젝트.
> 이 문서는 사람용(Notion)이자 Claude·코드용(repo) 단일 진실 소스(Single Source of Truth)다.
> 회사 ↔ 집 어느 기기에서 작업을 이어가든, 이 문서만 읽으면 맥락이 복원된다.

_최종 업데이트: 2026-06-17_

---

## 0. 한 줄 요약

> **"앱 하나"가 아니라 "앱을 빠르게 찍어내는 생산 라인"을 만든다.**
> 공용 디자인 시스템 + 공용 인프라 위에, 기능이 서로 다른 작은 유틸 앱들을 패시브 수익원으로 양산한다.

---

## 1. 개발자 프로필 (왜 이 전략인가)

- Unity로 게임 출시 경험 (1인)
- Android(Kotlin)로 앱 출시 경험 (1인)
- 이후 Android 실무 → Flutter 실무 → **현재 Flutter 개발자**
- 강점: **기획→출시 완주 경험**, **게임적 감각**, **크로스플랫폼(Flutter)**
- 핵심 통찰: **"만들기"는 병목이 아니다.** (이미 능력 충분 + AI 보조) 진짜 병목은 **유통(공짜 유입)** 이다.

## 3. 방향 결정: 유틸 앱 양산

검토한 후보와 결론:

| 방향 | 공짜 유입 통로 | 패시브 적합 | 결론 |
|---|---|---|---|
| **틈새 유틸 앱 양산** | 앱스토어 검색(ASO) | ✅✅ | **채택** |
| 하이브리드 캐주얼 게임 | 바이럴/추천/광고비 | ❌ | 보류 (유입 0 위험) |
| 콘텐츠 웹+광고(SEO) | 구글 검색 | △ | 보류 (Flutter 안 씀) |

- **게임 양산 비추 이유:** AI는 게임 *코드*는 잘 짜지만 *재미(손맛·밸런싱·리텐션)*는 못 채운다. 그런데 돈은 바로 그 "재미"에서 나온다. 게다가 게임은 *검색 유입이 거의 없다*.
- **게임 감각은 버리지 않는다:** "유틸의 탈을 쓴 게임적 경험"(안 지루한 트래커, 손맛 나는 타이머)으로 리텐션 무기로 쓴다.

### 중복 앱(clone) 리젝 기준 — 정리됨

- **시각적 유사성은 트리거 아님.** (Material끼리 닮는 건 정상)
- 진짜 트리거: **기능적·구조적 중복** = 같은 앱을 색/이름만 바꿔 여러 개(reskin), 자동생성기 양산, 최소기능 미달.
- 근거: Apple 4.3(Spam)·4.2(Minimum Functionality), Google "Spam & Repetitive Content".
- **우리 전략은 "다른 앱을 빠르게"라 안전.** 공용 디자인 시스템 공유는 자산이지 리스크 아님.
- 안전 수칙: ① 앱마다 진짜 다른 기능+실질 가치 ② 색만 바꿔 쪼개지 말 것 ③ 스토어 설명·스크린샷은 앱마다 직접 작성 ④ 코드 공유는 마음껏.

## 4. 아키텍처: 양산 파이프라인

### 4.1 공유할 두 개의 층

```
① 디자인 시스템 → 버튼·카드·테마·토큰 (보이는 것)
② 앱 인프라     → 광고·분석·로컬저장·설정화면·페이월·리뷰유도 (안 보이는데 매번 똑같은 것)
```

### 4.2 모노레포 구조 (Melos)

```
app_factory/                      ← 모노레포 (git 1개)
├── CLAUDE.md                     ← Claude 자동 로드 프로젝트 지침
├── docs/PLANNING.md              ← (이 문서)
├── packages/
│   ├── design_system/            ← ① 토큰·테마·공용 위젯 (브랜드 무관)
│   │   ├── tokens/               (spacing, radius, motion, typography 스케일)
│   │   ├── theme/                (BrandConfig → ThemeData 빌더)
│   │   └── components/           (AppButton, AppCard, AppDialog, EmptyState…)
│   └── core/                     ← ② 앱 인프라
│       ├── ads/                  (AdMob 래퍼)
│       ├── storage/              (Hive/Isar 래퍼)
│       ├── analytics/
│       ├── purchases/            (인앱결제·페이월)
│       └── common_screens/       (설정·약관·정보 화면)
├── apps/
│   ├── dday/                     ← 얇은 앱. 브랜드 설정 + 고유 화면만
│   └── subscription/             ← 70% 복붙
└── gallery/                      ← Widgetbook (디자인 시스템 전시장)
```

각 앱은 `path` 의존성으로 패키지를 끌어다 쓴다 → design_system 한 줄 고치면 모든 앱 즉시 반영.

### 4.3 핵심 설계 원칙 🔑 "브랜드는 주입, 컴포넌트는 무지"

- `design_system`의 위젯은 색·폰트를 *모른다*. `Theme.of(context)`만 본다.
- 각 앱은 시작 시 자기 **`BrandConfig`(seed 컬러, 폰트, radius, vibe)** 만 주입.
- 결과: **버튼 코드 1벌 → 앱마다 다른 외모.** (D-Day=말랑 핑크, 구독관리=단정 블루)
- 목적은 "리젝 위장"이 아니라 **각 앱의 정체성 부여**.

### 4.4 도구 한 장 요약

| 도구 | 역할 |
|---|---|
| Melos | 모노레포 + 패키지 일괄 관리 (지휘자) |
| design_system 패키지 | 브랜드 무지한 공용 위젯·토큰 |
| core 패키지 | 광고·저장·분석·공용화면 (안 보이는 절반) |
| BrandConfig 주입 | 코드 1벌 → 앱마다 다른 피부 |
| Widgetbook | 컴포넌트 전시장(쇼룸) — DS 독립 개발 |
| Mason brick | 새 앱 1줄 생성 = 생산 라인 |

> ⚠️ 규율: path 의존성이라 design_system을 깨면 모든 앱이 동시에 깨진다. 안정되면 시그니처를 함부로 바꾸지 않는다.
> 💡 처음부터 6개 다 깔지 않는다. D-Day는 **모노레포 + 패키지 2개**면 충분. Widgetbook·Mason은 **두 번째 앱부터** 빛난다.

### 4.5 서버 비용 = $0

- D-Day·구독관리 모두 데이터가 폰 안에만 있어도 충분 → 서버비 0 → 진짜 패시브.
- 백업·동기화는 나중에 Supabase로, 그때 **유료 기능**으로 팔 수 있음.

## 5. 첫 제품: D-Day 카운터

- **왜 1번:** 기능 단순 → 완주 확률 ↑. 첫 앱의 진짜 목적은 **출시 파이프라인 까기**(광고·스토어 등록·심사·분석·위젯·다국어).
- **2번:** 구독 관리 트래커 — 위젯·광고·심사 노하우 70% 재활용.
- 핵심 기능 후보: 날짜 등록 → D-Day 카운트, **홈 위젯**, 커스텀 테마, 알림.
- 차별화: 위젯 완성도 + 테마 감각.

```
1단계: D-Day 출시      ← 파이프라인 까는 게 진짜 목적
2단계: 구독관리 출시   ← 재활용으로 가속
```

> ⚠️ 1인 개발 망하는 1순위 = 기능 욕심. MVP는 "출시에 꼭 필요한 최소"만.

## 6. 연속성 (회사 ↔ 집)

- 이 Claude **세션/메모리는 로컬** → 다른 기기에 안 따라옴.
- 해결: **프로젝트가 스스로 맥락을 들고 다니게.**
  - 코드 → GitHub push/pull
  - 결정·맥락 → repo의 `CLAUDE.md` + 이 `docs/PLANNING.md`
  - 사람용 대시보드 → Notion (이 문서를 마크다운 import)
- 다른 기기에서 repo pull + Claude Code 실행 → `CLAUDE.md` 자동 로드 → 맥락 즉시 복원.
